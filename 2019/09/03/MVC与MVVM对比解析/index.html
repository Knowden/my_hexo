MVC与MVVM对比解析

发表于 2019-09-03 分类于 Server

新搞了这个博客，试试效果咋样

今天花了点时间好好研究了一下MVC和MVVM两种设计模式，以及他们在前后端的使用，还是挺有收获的，所以做一下记录

MVC模式

MVC模式就是传统的模型、控制器、视图模式

重点说一说这个模式在前端和后端的使用吧

这个设计模式主要是使用在JS脚本的开发中，目的是为了让JS和HTML充分解耦，力争在HTML代码中不要出现JS代码

简单说一下这三个部件的作用：

  1. 控制器：主要作用是控制整个逻辑的执行流程，像是一个指挥官，也正如其名，控制器

  2. 模型：模型是数据的来源，模型的功能是获取数据和提供数据给控制器，由控制器将其发送给视图

  3. 视图：很简单，就是我们看到的视图，通过对DOM的操作，可以根据模型提供的数据动态修改HTML代码（注意，这和服务器上跑的模板引擎不是一个东西，这里用的是纯原生JS功能）

清晰了这三个概念，MVC的模式在前端的使用也就清晰了

  • 首先，模型获取了数据，调用控制器的某个方法，通知控制器自己的数据发生了变化

  • 然后，控制器按程序逻辑将模型传给相应的视图

  • 最后，视图层的代码根据传入的模型，对HTML元素进行动态修改

以上就是MVC在前端的执行流程

下面说一下MVC在后端中的使用

跟在前端类似，这三层执行的工作基本不变，只不过视图层往往使用模板引擎直接在服务器渲染HTML，这样加大了服务器的负担，不是很好的设计，前后端的耦合度也会较高，以后的文章中将介绍的前后端分离技术会解决这个问题

MVVM模式

这个模式仅用于前端，代表框架有Vue，主要解决的是MVC中依然存在视图直接操作模型的情况

这个模式的核心是数据的双向绑定，也就是视图中的DOM和模型中的数据，通过中间的ModelView来进行了一个双向绑定

当用户操作界面引发DOM变化的时候，模型中对应的数据会发生变化

当模型中的数据发生变化的时候，视图中的DOM对象会同步变化

这样之前MVC模式中大量的DOM更新操作和数据同步操作都被省去了，耦合度大大降低
